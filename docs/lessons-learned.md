## 1. テストが CI で落ちる ── ローカルでは通るのに

### 状況

ローカル（Windows）では全テストが通るのに、CI（GitHub Actions / Ubuntu）で特定のテストが失敗する。

### 原因

**画面サイズの違い。** CI 環境のテストは 800x600 のデフォルトサイズで実行される。ローカルの Windows 環境はそれより大きい画面で実行されるため、ドロワー内のナビゲーション項目が全て画面内に収まるが、CI では一部が画面外にはみ出す。

### 解決策

ドロワー操作を共通ヘルパーに抽出し、スクロール対応を自動化した。

```dart
// Before: 画面サイズに依存する直接タップ
await tester.tap(find.text('統計').last);

// After: スクロール対応の共通ヘルパー
await navigateViaDrawerInTest(tester, '統計');
```

### 教訓

> **テストは「最も制約の厳しい環境」を基準に書く。**
> ローカルで通ることは品質の保証にならない。CI 環境の制約（画面サイズ、メモリ、ネットワーク）を常に意識する。

## 2. ダイアログが操作をブロックする ── 新機能追加の副作用

### 状況

初回アクセス時のオンボーディングダイアログを追加した後、既存のテスト7件が一斉に失敗した。

### 原因

オンボーディングダイアログが `barrierDismissible: false`（背景タップで閉じない）かつ `barrierColor: Colors.black87`（背景を暗くする）で画面全体を覆うため、テストがハンバーガーメニューをタップできなくなった。

### 解決策

テストの SharedPreferences 初期値に `onboarding_completed: true` を追加し、テスト実行時はオンボーディングをスキップするようにした。

```dart
// Before
SharedPreferences.setMockInitialValues({});

// After
SharedPreferences.setMockInitialValues({'onboarding_completed': true});
```

### 教訓

> **新しいダイアログやモーダルを追加するときは、既存テストへの影響を必ず確認する。**
> 特に `barrierDismissible: false` のダイアログは、テスト環境で他の操作を完全にブロックする。
> テスト用の初期値は共通ヘルパーで一元管理し、新機能追加時にここだけ更新すれば済むようにする。

## 3. シングルトンの罠 ── テスト間で状態がリークする

### 状況

`testWidgets` で複数のテストを実行すると、前のテストで遷移したガントチャートページの状態が次のテストに引き継がれ、ボトムナビゲーションが見つからない `RangeError` が発生した。

### 原因

`GoRouter` がモジュールレベルのシングルトンとして定義されており、テスト間でルーター状態がリセットされない。前のテストがガントチャートページ（ボトムナビなし）で終了すると、次のテストもそのページから開始される。

```dart
// app.dart — モジュールレベルのシングルトン
final _router = GoRouter(routes: [...]);
```

### 解決策

1. `tapBottomNav` ヘルパーに、ボトムナビが存在しない場合はドロワー経由でホームに戻る自動復旧ロジックを追加
2. テストの冒頭でドロワー経由の遷移を使い、GoRouter の状態に依存しない設計にした

```dart
Future<void> tapBottomNav(WidgetTester tester, int index) async {
  // ボトムナビがない場合は自動復旧
  if (find.byType(NavigationDestination).evaluate().isEmpty) {
    await navigateViaDrawer(tester, 'ダッシュボード');
  }
  await tester.tap(find.byType(NavigationDestination).at(index));
  await tester.pumpAndSettle();
}
```

### 教訓

> **シングルトンはテストの敵。** グローバルな状態を持つオブジェクトは、テスト間で状態がリークする。
> 完全な解決はシングルトンを避けることだが、既存コードを変更できない場合は「テストヘルパーで状態を復旧する」防御的アプローチが有効。

## 4. UI の重なり問題 ── 見えているのにタップできない

### 状況

ドロワーの「統計」テキストが画面上で見えているのに、テストの `tester.tap()` が「would not hit test on the specified widget」と警告して失敗する。

### 原因

ドロワーの `Column` レイアウトで、`Expanded` 内の `ListView`（スクロール可能エリア）と、その下の固定配置の「使い方」`ListTile` が物理的に重なっていた。Flutter のヒットテストでは、上に配置されたウィジェットが優先されるため、ListView 内の「統計」がタップ不能になる。

```
┌─────────────┐
│  ListView    │  ← 「統計」はここの最下部
│  ...         │
│  統計        │  ← 物理的にはここにあるが...
├─────────────┤
│  使い方      │  ← このウィジェットが上に重なってタップを奪う
│  設定        │
└─────────────┘
```

### 解決策

「使い方」ボタンを固定領域から ListView 内に移動し、スクロール可能にした。

### 教訓

> **Flutter のヒットテストは z-order（描画順序）に従う。**
> 「見えている」ことと「タップできる」ことは別問題。
> `Expanded` + `ListView` と固定配置ウィジェットを組み合わせるときは、重なり領域に注意する。
> テストの `tap()` 警告メッセージには、実際にどのウィジェットがヒットしたかが表示されるので、それを手がかりに原因を特定する。

## 5. 日時のテスト ── 1ミリ秒の差で落ちる

### 状況

招待コードの有効期限テストで `Expected: <60>`, `Actual: <59>` が発生。60日間有効なはずなのに残り日数が59日と判定される。

### 原因

テスト内で `DateTime.now()` を SharedPreferences に保存した後、`getStatus()` を呼ぶまでにミリ秒単位の時間が経過する。その結果、有効化日時と現在時刻の差が微妙にずれ、`remainingDays` の切り捨て計算で 60 ではなく 59 になる。

```dart
// 保存時: 2026-03-17 10:00:00.000
await service.activate('CODE', config);
// 検証時: 2026-03-17 10:00:00.005 ← 5ミリ秒経過
final status = service.getStatus();
// remainingDays = 59（ミリ秒の差で1日分ずれる）
```

### 解決策

厳密な一致ではなく `closeTo` マッチャーで許容範囲を持たせた。

```dart
// Before: 厳密一致（タイミングで落ちる）
expect(status.remainingDays, 60);

// After: ±1日の許容範囲
expect(status.remainingDays, closeTo(60, 1));
```

### 教訓

> **テストで `DateTime.now()` を直接使うと、タイミング依存の不安定なテストになる。**
> 対策は3つ:
> 1. `closeTo` で許容範囲を持たせる（簡単）
> 2. `Clock` クラスを注入して固定時刻を使う（確実）
> 3. テスト内で固定の `DateTime` を使い、`DateTime.now()` を避ける（最も安全）

## 6. CI ワークフローの競合 ── 2つのジョブが同時にプッシュ

### 状況

SonarCloud ワークフローが `git push` で失敗する。エラーメッセージは「Updates were rejected because the remote contains work that you do not have locally」。

### 原因

`sonarcloud.yml` と `integration_test.yml` が同じプッシュイベントで同時に起動し、両方がレポートファイルを `main` ブランチにコミット＆プッシュしようとする。先にプッシュした方が成功し、後からプッシュした方が競合で失敗する。

```
push → sonarcloud.yml（レポートコミット＆プッシュ）──→ 成功
push → integration_test.yml（レポートコミット＆プッシュ）──→ 競合で失敗
```

### 解決策

これはコードの問題ではなく、ワークフロー設計の問題。レポートの自動コミットは `[skip ci]` タグ付きで行われるため、連鎖的なワークフロー起動は防止済み。競合が発生した場合は再実行で解消される。

根本的には、レポートを別ブランチに保存するか、Artifacts として保存する方法がある。

### 教訓

> **CI で自動コミットを行うワークフローが複数ある場合、同時実行の競合を考慮する。**
> 対策の選択肢:
> 1. `concurrency` グループで排他制御する
> 2. レポートを Artifacts に保存し、リポジトリへのコミットを避ける
> 3. 競合時のリトライロジックを追加する（`git pull --rebase && git push`）

## 7. モバイル対応の落とし穴 ── PC では問題ないのに

### 状況

iPhone 15 でアプリを開くと、チュートリアルのメッセージが日付選択 DatePicker に被って操作できない。フィードバックダイアログでは縦帯（オーバーフロー表示）が発生。

### 原因

1. **メッセージの被り**: チュートリアルバブルの位置計算が PC 画面サイズを前提としており、モバイルの狭い画面では DatePicker と重なる
2. **縦帯**: フィードバックダイアログ内のドロップダウンに `SizedBox(width: 480)` が固定値で指定されており、モバイルの画面幅（約375px）を超過

### 解決策

1. ダイアログが開いている時はメッセージを画面下部に移動（`_DialogBubble` の導入）
2. 固定幅を `ConstrainedBox(maxWidth: 480)` に変更し、画面幅に応じて縮小されるようにした

### 教訓

> **Web アプリでも、モバイルブラウザからのアクセスを常に想定する。**
> 特に注意すべき点:
> - 固定幅（`width: 480`）ではなく最大幅（`maxWidth: 480`）を使う
> - `MediaQuery.of(context).size` で画面サイズを取得し、レイアウトを分岐させる
> - 実機テスト（または Chrome DevTools のデバイスモード）で確認する

## 8. 外部 API のキー管理 ── ハードコードの誘惑

### 状況

Google Apps Script のエンドポイント URL や GitHub Gist ID をソースコードに直接記述している。Stripe のシークレットキーをどこに保存すべきか判断が必要だった。

### 判断

| 情報 | 保存場所 | リスク評価 |
|------|---------|----------|
| Gist ID | ソースコード内 | 低（Secret Gist、URL を知らない限りアクセス不可） |
| GAS エンドポイント URL | ソースコード内 | 低（POST のみ受付、不正リクエストは無害） |
| Stripe 公開キー | ソースコード内（テスト用） | 低（公開キーは公開前提） |
| Stripe シークレットキー | GAS 内 | 中（GAS のソースは非公開） |

### 教訓

> **「ハードコードは悪」は原則だが、状況によっては許容される。**
> 判断基準:
> 1. そのキーが漏洩した場合の被害は何か？
> 2. キーのローテーション（更新）は容易か？
> 3. 環境変数やシークレット管理サービスを使うコストは見合うか？
>
> 個人開発では「過剰なセキュリティ設計」と「実装コスト」のバランスが重要。
> ただし、**本番のシークレットキーは絶対にソースコードにコミットしない**ことは守る。

## 9. カテゴリ名変更の波及 ── 1行の変更が7テストを壊す

### 状況

フィードバックのカテゴリ名を `'不具合報告'` → `'不具合'` に変更したら、テストが1件失敗した。さらに別の修正で `'改善要望'` 等のカテゴリを変更した際にも、テスト内のハードコードされた文字列が不一致になった。

### 原因

テスト内でカテゴリ名を文字列リテラルで直接参照していた。ソースコード側のカテゴリ名を変更しても、テスト側は自動的には追従しない。

```dart
// テスト側（ハードコード）
expect(body['category'], '不具合報告');  // ← ソース変更に追従しない

// ソース側（変更後）
bugReport('不具合'),  // ← '報告' を削除
```

### 解決策

テストでカテゴリ名を直接参照する代わりに、enum の `.label` プロパティを使う。

```dart
// After: enum を参照
expect(body['category'], FeedbackCategory.bugReport.label);
```

### 教訓

> **テストでマジックストリングを使わない。** ソースコードで定義された定数や enum を参照することで、
> 変更時の不整合を防ぐ。テストは「実装の変更に強い」書き方を心がける。
>
> これは「テストが仕様を表現する」という原則にも通じる。
> テストが壊れるのは「仕様が変わった時」だけであるべきで、「表記が変わった時」ではない。

## 10. チュートリアルの z-order 問題 ── 画面の重なり順序

### 状況

チュートリアルのスポットライト（グレーアウト + 穴あき）がダイアログの上に表示されてしまい、ダイアログの操作ができない。逆に、チュートリアルメッセージがダイアログの裏に隠れて見えない。

### 原因

Flutter の描画順序（z-order）の理解不足。

```
MaterialApp.builder   ← 最上位（TutorialOverlay はここ）
  Navigator overlay   ← ダイアログはここに描画される
    Scaffold          ← 通常のページ
      Stack           ← TutorialSpotlight はここ
        Body
```

`TutorialSpotlight`（Scaffold の Stack）はダイアログ（Navigator overlay）より下に描画されるため、ダイアログの上にスポットライトを表示できない。

### 解決策

2層アーキテクチャを採用:

- **TutorialOverlay**（MaterialApp.builder レベル）: ダイアログの上に表示。完了ダイアログやダイアログ上のメッセージを担当
- **TutorialSpotlight**（Scaffold の Stack レベル）: 通常ページのスポットライト・メッセージを担当。ダイアログが開いている時は自動的に非表示

### 教訓

> **Flutter の Widget ツリーは描画順序に直結する。**
> 「何を」「どのレイヤーに」配置するかを設計段階で決める。
> 特にオーバーレイ系の UI（チュートリアル、ツールチップ、トースト）は、
> `MaterialApp.builder` / `Overlay` / `Stack` のどこに配置するかで挙動が大きく変わる。

## 11. 「動くコード」と「正しいコード」の違い

### 状況

体験版の制限ロジックで、夢の数を「夢テーブルのレコード数」でカウントしていたが、チュートリアルで作成した夢が中途半端に残り、インデックスが `0, 0, 0, 1` のように不正になることがあった。

### 原因

チュートリアル終了時のデータクリーンアップが不完全で、「作成途中の夢」がDBに残る場合があった。制限チェックはレコード数を数えるだけなので、不正なデータも有効な夢としてカウントされる。

### 解決策

1. チュートリアル終了時に「データを保持するか削除するか」をユーザーに選択させるダイアログを追加
2. 削除を選択した場合は、チュートリアルで作成した夢・目標を確実に削除

### 教訓

> **「正常系で動く」だけではリリースできない。**
> 以下を常に考える:
> - ユーザーが途中で操作を中断したらどうなるか？
> - 不正なデータが残ったらどうなるか？
> - エッジケース（0件、上限、境界値）でどうなるか？
>
> 特にチュートリアルのような「ガイド付きフロー」は、途中離脱のケースを必ず設計する。

## 12. 個人開発で学んだこと

### AI ペアプログラミングの効果

Claude Code との協働で実感した効果:

| 作業 | AI なし（推定） | AI あり（実績） | 効果 |
|------|---------------|----------------|------|
| アーキテクチャ設計 | 数日 | 数時間 | 選択肢の提示と比較が即座 |
| テストコード作成 | ソースと同等以上 | ソースの20%程度 | テストパターンの自動提案 |
| バグ修正 | 原因特定に時間 | ログから即座に特定 | CI ログの解析が高速 |
| ドキュメント作成 | 後回しになりがち | 開発と同時に作成 | コンテキストが新鮮なうちに文書化 |

### 「完璧」を求めない

個人開発で最も重要な判断は「どこで妥協するか」だった。

- **GitHub Gist でリモート設定** ← Firebase Remote Config は過剰
- **Google Apps Script でバックエンド** ← 自前サーバーは不要
- **SharedPreferences で招待コード管理** ← サーバーサイド認証は後回し

これらは全て「今の規模では十分」という判断。スケールする必要が出た時に改善すればよい。

### 継続的な改善の仕組み

コードを書くだけでなく、**改善が自動的に回る仕組み**を作ることが重要だった:

1. **プッシュ → テスト自動実行** ← 壊れたらすぐ分かる
2. **プッシュ → SonarCloud 解析** ← 品質劣化が可視化される
3. **プッシュ → 自動デプロイ** ← 手動作業ゼロでリリース
4. **レポート自動保存** ← Claude Code が次回の修正に活用

この仕組みがあるからこそ、個人開発でも高い品質を維持できた。

### 最後に

> **個人開発は「全部自分でやる」ことではない。**
> 使えるツール・サービス・AI を最大限に活用し、
> 自分は「何を作るか」「なぜ作るか」に集中する。
> 技術は手段であり、目的は「ユーザーに価値を届けること」。
